The four protocols decide in genuinely different ways — that heterogeneity is the comparison problem. (1) The four protocols sets up the problem; (2) Figure A — the harness is the solution (one fixed engine, the protocol is the only swappable part); (3) Figure B — the event loop is how it runs. Read in order.

1.The four protocols we compare the problem

Each protocol reaches a decision by a different rule, and the four do not even produce the same kind of decision. This table is the reference the harness figure points back to; the exact rules stay here so the figures can stay light.

ProtocolHow it decides (plain)Decides exactly when (precise)Leader?One "decision" (ACU) =
PBFT A leader proposes; replicas vote in two rounds. A replica commits: it gathers 2f+1 PREPARE, then 2f+1 COMMIT — two quorums, not one. Yes (rotating primary) one committed block
Casper FFG Validators attest each epoch; finality takes two rounds. A checkpoint is final when it is justified and its direct child is justified — two consecutive ≥ ⅔-stake links, ≥ 2 epochs. One ≥ ⅔ link alone is only justification. Yes (slot proposer) one finalized checkpoint (+ its ancestors)
Snowman Repeatedly poll K random peers until confident. Accepted after β consecutive polls that each clear a confidence threshold (α_c); preference flips at a separate threshold (α_p). No (leaderless) one accepted block
Narwhal + Tusk A DAG mempool spreads data; Tusk then orders it. An anchor commits once enough round-(r+1) certificates reference it. Ordering adds zero messages — but rides on ≈ r·n² mempool messages per round. Yes (anchor) one committed anchor (a batch / sub-DAG)
f = the most faulty nodes a protocol tolerates. ACU (atomic commit unit) = the per-protocol thing one decision commits, shown in the last column. Because a PBFT block, an FFG checkpoint-plus-ancestors, a Snowman block, and a Narwhal anchor-batch are different sizes of thing, every rate is reported per ACU so the four land on one scale.
The takeaway that sets up Figure A: four different deciding rules, two different decision shapes, one protocol leaderless, one with two layers — so the comparison can only be fair if a single fixed engine runs them all and a reconcile step normalizes their output.

What it shows

  • Input is one cell of the experiment matrix — the knobs we vary (protocol, n, network, adversary, workload, seed) are the independent variables the research questions sweep.
  • The engine is fixed, but that alone is not "fairness." A shared Builder / Scheduler / Network / Logger removes harness confounds, so any difference comes from the protocol logic — but making four different decisions comparable is the separate reduce + reconcile step, and that is where commensurability is established, under stated conventions (not in the engine being shared).
  • The adversary is not a separate component — it attaches to a node and alters that node's outgoing messages; its effect is protocol-specific (there is no leader to disrupt in Snowman). Drawn as an input knob for brevity.
  • Output is one comparable data point → the RQs, all per ACU.

3.Figure B — One turn of the event loop the mechanism

A zoom into the Scheduler. The engine repeats one cycle until the run ends: take the next-soonest event, advance the clock to it, let a node react, and put whatever that produces back on the queue.

flowchart TD
    Q["EVENT QUEUE
min-heap ordered by (time t, node, sequence)"]
    N["NODE — runs its protocol logic for this event"]
    NET["NETWORK — turns a send into a future delivery
at time t + delay (loss / partitions applied here)"]
    ENQ(["future event
(a delivery, or a timer)"])
    STOP["STOP — the run ends when:
nothing left to do · the time limit is reached · the caller's stop condition fires"]

    Q -- "pop next, advance clock" --> N

The three ways a run ends are recorded as quiescence (nothing left to do), deadline (time limit reached), and predicate (caller's condition). A deadline stop is not automatically a failure — whether it is a real liveness failure is judged separately, by whether the target decision was reached before the deadline.

What it shows

  • The clock jumps, it does not tick. Each turn the scheduler takes the earliest event and sets the virtual clock straight to its time t, skipping idle gaps. (This is about efficiency, not yet reproducibility.)
  • Why a run replays identically. Determinism rests on three things together: the queue's total order (t, node, sequence) (so even same-time events have one order), per-node RNGs seeded from the run seed, and the rule that nothing reads a wall clock or iterates in a nondeterministic order. Clock-jumping alone would not guarantee it.
  • One feedback path drives everything. A node only reacts by sending a message, setting a timer, or emitting a result (decided / halted, which the Logger records). Sends and timers become future events; there is no other source of work.
  • The node is generic here — this loop is identical for all four protocols; what runs inside is the swappable protocol logic from the table.

How the three pieces connect

  • The table is what the harness's swappable slot holds — four heterogeneous deciding rules.
  • Figure A's SCHEDULER box is the loop drawn in Figure B — A says "there is a run loop," B shows one turn.
  • The decided / halted results a node emits in Figure B are what the Logger captures and Figure A's reduce + reconcile step turns into the per-ACU results.csv row.
§3.2 prose checklist — these are obligations, not figure elements. The figures defer the honest version of each claim to the prose; §3.2 is not complete until each is discharged (and cited against the wiki):
  • Commensurability. Where §3.2 first calls the comparison fair, cite the metric-reconciliation layer and state the conventions it assumes (per-ACU denominators per the table, Narwhal layer split, Snowman rescaling) — and concede it makes the four plottable on one scale under conventions, not literally the same event.
  • No compute / crypto cost. Name it as a threat to validity; it favours signature-heavy protocols (PBFT O(n²), Narwhal mempool). Bound which verdicts (overhead, latency) it could bias.
  • FFG ↔ network coherence. State the guard E[delay] ≤ slot_duration/4 and that the runner refuses incoherent pairings, so delay sweeps cannot silently produce degraded-finality "Casper FFG" numbers.
  • Workload realism. State that the load is pre-seeded / closed-loop and name its arrival process, conflict rate, offered rate.
  • Scaling range. n = 4..25 (f = 1,2,3,5,8) is a stated limit on the scaling claim (RQ3); note Snowman's n=4 exclusion so a missing plot point is explained.
  • Snowman rescaling validity. Defend that a per-n-rescaled Snowman is still one protocol across the scaling sweep, not four parametrizations.
  • Deadline vs liveness failure. Spell out which output column lets a reader reconstruct a truncated-healthy run from a real liveness failure, so RQ4 success rates are not contaminated.
  • Reliability family. §3.2 must define agreement vs termination and the safety-violation rate (ε for Snowman) the OUTPUT box only names.
